## 1. 기획 — 왜 만들었나

### 해결하려는 문제

Slack 헬프데스크 채널에 "사업자등록증 받을 수 있나요?", "재직증명서 어디서 구하나요?" 같은 문서 요청 메시지가 반복적으로 들어온다. 담당자가 매번 수동으로 답변하면 응답 지연과 업무 중단이 발생한다.

### 해결 방식

Slack 메시지를 실시간으로 감지해, 문서 요청이라고 판단되면 Notion 자료실에서 가져온 실물 파일을 해당 스레드에 자동으로 달아준다.

### 봇이 하는 일 (한 줄 요약)

헬프데스크 채널의 문서 요청 메시지를 감지하고, 등록된 실물 파일을 스레드로 즉시 전송한다.

### 봇이 하지 않는 일

- 자유 대화 응답 (Claude AI 미사용)
- 문서 직접 생성
- 등록되지 않은 요청 처리

## 2. 프롬프팅 — 봇은 어떻게 요청을 감지하나

이 봇은 AI 모델을 쓰지 않는다. 순수 키워드 매칭 방식으로 동작한다. 빠르고, 비용이 없으며, 동작 예측이 가능하다.

### 감지 조건

메시지 안에 아래 두 가지가 **동시에** 포함되어야 봇이 반응한다.

**조건 1 — 요청 동사 (REQUEST_VERBS)**

```
받, 주세요, 보내, 구할, 확인, 필요, 어디, 찾아줘
```

**조건 2 — 등록된 문서명 또는 별칭 (aliases)**

`knowledge/documents.json` 파일에 각 문서마다 alias(별칭) 목록이 있다.
예: "사업자등록증"은 "사업자", "사업자증", "사업자등록" 등의 별칭도 함께 등록된다.

두 조건이 모두 충족될 때만 봇이 답변을 전송한다. 하나만 있으면 무시한다.

### 모호한 키워드 처리

혼동 가능성이 있는 키워드(예: "신고증", "영화")가 포함된 경우, 봇은 자동 답변 대신 "어떤 문서를 찾으시나요?" 형식으로 추가 입력을 유도한다.

### 테스트 메시지 형식

```
test: 사업자등록증 보내주세요
```

`test:` 접두사가 붙으면 헬프데스크 채널에 답변하지 않고, GA 팀 채널에 승인 카드를 먼저 보낸다.

## 3. 코딩 — 파일 구조와 역할

### 프로젝트 폴더 구조

```
doc-bot/
├── main.py                      # 봇 진입점. Slack 이벤트 수신 시작
├── config.py                    # 채널 ID, 환경변수 등 설정 상수 모음
├── requirements.txt             # Python 패키지 목록
├── refresh_docs.py              # Notion 자료실에서 파일 다운로드
├── Procfile                     # Railway 실행 명령 (web: python main.py)
├── railway.toml                 # Railway 배포 설정
├── agents/
│   └── doc_request_agent.py    # 문서 요청 감지, 답변 생성, 파일 업로드 담당
└── knowledge/
    ├── documents.json           # 등록된 문서 목록 (별칭 + 파일 경로)
    └── files/                   # 로컬에 캐시된 실물 파일
```

### 각 파일이 하는 일

| 파일 | 역할 |
|------|------|
| `main.py` | Slack 연결하고 메시지 이벤트를 에이전트에 넘김. startup_check()로 상태 확인 후 봇 기동 |
| `config.py` | 채널 ID, 환경변수 키 등 수정이 잦은 상수를 한 곳에 모아둠 |
| `refresh_docs.py` | Notion API로 자료실 DB를 조회하고, 실물 파일이 있는 항목만 다운로드해 documents.json 갱신 |
| `doc_request_agent.py` | 메시지를 분석해 요청 여부 판단, 해당 문서 검색, 스레드에 답변 전송 |
| `documents.json` | 봇이 알고 있는 문서 목록. refresh_docs.py 실행 시 자동 갱신 |

### 환경변수

| 변수명 | 설명 |
|--------|------|
| `DOC_BOT_TOKEN` | Slack 봇 토큰 (xoxb-...) |
| `DOC_APP_TOKEN` | Slack 앱 토큰, Socket Mode용 (xapp-...) |
| `NOTION_TOKEN` | Notion API 통합 토큰 (secret_...) |
| `TEST_MODE` | `true`면 팀 채널에 승인 카드 발송, `false`면 헬프데스크에 자동 답변 |

### 로컬 실행 방법

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python refresh_docs.py   # Notion 파일 다운로드
python main.py           # 봇 실행
```

### Slack 앱 설정

OAuth 스코프: `channels:history`, `channels:read`, `chat:write`, `files:write`, `reactions:write`

이벤트 구독: `message.channels` (Socket Mode 활성화 필요)

채널 ID:

|------|----|

| GA 팀 채널 | C08823F52U8 |

## 4. 리뷰 — 설계 결정과 배경

"왜 이렇게 만들었나"를 기록한다. 나중에 수정할 때 맥락을 이해하고 판단하기 위한 내용이다.

### 결정 1. Claude AI를 쓰지 않는다

- 문서 요청은 패턴이 일정해 키워드 매칭으로 충분히 처리 가능
- API 비용 없음, 응답 빠름, 동작 예측 가능

### 결정 2. 자동 갱신 스케줄러를 제거했다

- APScheduler를 넣으면 편리하지만, 타이밍 오류·스케줄 충돌 등 예측하기 어려운 문제 발생 가능
- 팀원 유지보수 부담 최소화를 위해 수동 `python refresh_docs.py` 방식 채택
- 절차가 명확하고 실패 지점이 적다

### 결정 3. 실물 파일이 있는 문서만 등록한다

- Notion "문서 파일" 속성에 실제 파일이 첨부된 항목만 documents.json에 등록
- 링크만 있는 항목은 Notion 접근 권한 문제로 빈 화면이 뜰 수 있어 제외

### 결정 4. TEST_MODE로 프로덕션 영향을 분리했다

- 코드 변경 후 바로 헬프데스크에 오답변이 나가는 사고 방지
- 테스트 채널에서 먼저 확인 후 운영 전환

## 5. 디버깅 — 문제 해결 절차

봇이 이상하게 동작할 때는 아래 순서를 지킨다.

### 기본 원칙

**Rule 1. 로그를 먼저 읽는다, 수정은 나중이다**

현상 확인 → 로그 확인 → 가설 수립 → 수정 순서를 지킨다.

**Rule 2. 재시작은 근거가 있을 때 1회만 한다**

"이 변경이 반영되려면 봇을 재시작해야 한다. 이유는 환경변수가 기동 시점에만 로드되기 때문이다." 형식으로 이유를 말할 수 없으면 재시작하지 않는다.

**Rule 3. 가설은 하나씩 검증한다**

가설 A를 검증하기 전에 가설 B로 넘어가지 않는다.

### 로그 확인 방법

Railway 배포 환경:
```
railway.com 대시보드 → 프로젝트 선택 → Deployments → 최신 배포 → Logs 탭
```

로컬 실행 환경: 터미널에 직접 출력됨

### 봇 재연결 시 필수 확인

봇을 재시작하거나 재배포한 후, 로그에서 `startup_check()` 통과 여부를 확인한다.

`startup_check()`가 하는 일:
- Slack `auth.test`로 토큰 유효성 확인
- 봇에 부여된 스코프 확인
- 헬프데스크 채널에 포스팅 가능한지 확인

### 트러블슈팅 체크리스트

```
[ ] 로그를 확인했는가?
[ ] 에러 메시지를 직접 읽었는가?
[ ] 근본 원인을 특정했는가?
[ ] 재시작이 정말 필요한가?
[ ] 이 변경이 원인을 직접 해결하는가?
```

### 자주 발생하는 문제

| 증상 | 확인할 것 |
|------|-----------|
| 봇이 메시지에 반응하지 않음 | TOKEN 유효 여부, 채널에 봇이 초대되어 있는지 |
| 파일이 전송되지 않음 | `files/` 폴더에 해당 파일이 있는지, `files:write` 스코프 있는지 |
| 엉뚱한 문서를 답변함 | `documents.json`의 aliases가 너무 짧거나 겹치는지 |
| 테스트 메시지인데 헬프데스크에 발송됨 | `TEST_MODE=true` 환경변수 적용 여부 |

## 6. 배포 — Railway 클라우드 운영

### 왜 클라우드에 올렸나

로컬 PC에서 실행하면 PC가 꺼지거나 네트워크가 끊길 때 봇도 멈춘다. 24/7 운영을 위해 Railway로 이전했다.

### 배포 구조

```
GitHub에 코드 push → Railway 자동 감지 → 빌드 및 재배포
```

### 환경변수 관리

`.env` 파일은 GitHub에 올리지 않는다. 토큰은 Railway 대시보드에서 직접 입력한다.

```
Railway 대시보드 → 프로젝트 → Variables 탭
```

운영 환경 설정값:

```
DOC_BOT_TOKEN=xoxb-...
DOC_APP_TOKEN=xapp-...
NOTION_TOKEN=secret_...
TEST_MODE=false
```

### 배포 상태 확인

```
railway.com → 프로젝트 → Deployments
```

- 초록색: 정상 실행 중
- 빨간색: 배포 실패. Logs에서 원인 확인

## 8. 유지보수 — 새 문서 추가 방법

코드 수정 없이 4단계로 처리한다.

**1단계. Notion 자료실에 페이지 추가**

- 페이지 제목: 문서 공식 명칭
- "문서 파일" 속성: 실물 파일 첨부 필수 (없으면 봇이 등록하지 않음)

**2단계. Claude Code에 리프레시 요청**

```
리프레시 해줘
```

자동으로 `python refresh_docs.py`가 실행되고 documents.json과 files/ 폴더가 갱신된다.

**3단계. 동작 확인**

```
test: [새 문서명] 보내주세요
```

**4단계. 필요시 별칭 보완**

봇이 특정 표현을 못 잡으면 Claude Code에:

```
사업자등록증의 alias에 "사업자 등록" 추가해줘
```
